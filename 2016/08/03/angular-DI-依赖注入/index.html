angular DI(依赖注入)

发表于 2016-08-03   |  
依赖注入（DI）
DI是一个设计模式，处理组件如何获取依赖。
angular中 injector子系统专门负责创建组件，解析依赖，并且提供给请求的组件。

使用DI
DI在angular中广泛使用。你可以用它定义模块或者给模块提供run或者config块的时候。

service，directive，filter和animation这些组件都是可注入的工厂方法或者构造方法。 Controller通过一个构造函数定义。这个构造函数可以被注入 服务或 value 依赖组件 ，也可以以依赖的方式提供给其他组件。

run 接受一个函数，这个函数可以被注入 service, value, constant 组件。 config 接受一个函数，这个函数可以被注入 provider, constant 组件作为依赖。不能注入 service 或value到配置阶段。

工厂方法
定义 directive, service, filter都是通过工厂方法。这些工厂方法注册在模块上面。

angular.module('myModule', [])
factory('serviceId', ['depService', function(depService) { // ... }]) directive('directiveName', ['depService', function(depService) { // ... }])
filter('filterName', ['depService', function(depService) { // ... }]);

模块方法
我们可以指定在运行阶段和配置阶段 的函数。这些函数可以注入依赖。
angular.module('myModule', []) config(['depProvider', function(depProvider) { // ... }]) .run(['depService', function(depService) { // ... }]);

控制器
控制器是类或构造函数。他负责提供应用程序行为给模板里的声明性标记。
someModule.controller('MyController', ['$scope', 'dep1', 'dep2', function($scope, dep1, dep2) { ... $scope.aMethod = function() { ... } ... }]);

和服务不一样，控制器可以有多个实例。

而外的依赖，控制器可以拿到 $scope:控制器和dom有关联并且可以访问响应的scope。其他组件就只能访问$rootScope
解析：如果一个控制器作为路由的一部分进行实例化。任何路由解析到的值可以注入控制器。

依赖注释
angular通过注入器来调用 某个函数。你需要给这些函数标注以便注入器知道需要注入什么服务。

有三种方式来标注你的代码 使用行内数组的方式来标注(推荐)
用$inject 来标注 通过函数参数隐含的方式

行内数组方式 
    `someModule.controller('MyController', ['$scope', 'greeter', function($scope, greeter) {
  // ...
}]);`

通过$inject 属性
var MyController = function($scope, greeter) { // ... } MyController.$inject = ['$scope', 'greeter']; someModule.controller('MyController', MyController);

为什么要依赖注入
有三种方式可以让一个组件获得依赖

  1. 组件直接调用 new 操作符来创建一个依赖。
  2. 组件自己在全局变量里面寻找依赖。
  3. 组件可以在需要的时候，让依赖传进来。

前两项通过创建和查找引用不太好，因为这样会导致代码太僵硬。这会使你很难再修改依赖 。第三种才可行，因为他把寻找依赖的责任从组件里剥离了。

`function SomeClass(greeter) {
  this.greeter = greeter;
}

SomeClass.prototype.doSomething = function(name) {
  this.greeter.greet(name);
}`

上面这个例子，SomeClass不用关心创建或定位依赖，他只用简单的使用greeter就行了。

这很理想，但是他把获取依赖的责任推给了构建 someClass的那些代码。
要管理依赖的创建，每个angular应用程序都有一个注入器。这个注入器是一个服务定位者，他负责创建和查找依赖。

创建一个新的注入器，这个注入器可以提供定义在myMOdule上面的组件。

`var injector = angular.injector(['ng', 'myModule']);
var greeter = injector.get('greeter');`

请求依赖解析解决了硬编码的问题。但是也意味着应用程序需要被传进来一个注入器。

当angular编译模板的时候，

`



function MyController($scope, greeter) {
  $scope.sayHello = function() {
    greeter.greet('Hello World');
  };
}


injector.instantiate(MyController);`

这是最好的结果。程序代码只用简单的声明依赖，而不用处理注入。这也没有违反lod原则。

lod原则：面向对象中，对象对其他地方知道的越少越好。
